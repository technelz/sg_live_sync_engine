# Security Group Live Sync / Reconciliation Engine

This project contains a dependency-aware AWS Security Group live synchronization and reconciliation tool.

The current version performs a **live source-to-live target comparison**. It reads both source and target security groups directly from AWS using an `--info-file` configuration file, compares the source-required state against the target, creates missing target security group shells, remaps security-group-to-security-group references, applies missing rules and tags, and then validates the final target state after apply mode.

This tool is designed for disaster recovery (DR) parity use cases where a target VPC must contain the required security group structure and rules from a source VPC while still allowing target-side environment-specific rules.

> Current behavior: this tool runs in **source-required-only mode** by default.  
> Extra ingress or egress rules in the target account are tolerated and are not treated as drift unless strict extra-rule visibility is intentionally enabled.

## What This Tool Does

The engine:

- Reads source security groups live from AWS.
- Reads target security groups live from AWS.
- Uses an info/config file to load source and target profile, region, and VPC details.
- Matches source SGs to target SGs using strict and optional legacy matching logic.
- Detects missing source-required ingress and egress rules.
- Creates missing target SG shells when needed.
- Handles dependency cycles by creating SG shells before applying rules.
- Remaps source SG references to matching target SG IDs.
- Adds missing source-required rules to the target.
- Syncs tags where applicable.
- Produces a Terraform-like execution plan.
- Writes a JSON report after each run.
- Tracks rollback actions for reversible apply operations.
- Performs post-apply validation after `--yes` mode by re-reading the target security groups and rebuilding the reconciliation plan.
- Adds a `post_apply_validation` section to the JSON report showing whether the target converged after remediation.

## Current Reconciliation Policy

This tool currently follows a **source-required-only** policy.

That means:

| Scenario | Result |
|---|---|
| Source SG exists in target | OK |
| Source SG missing in target | Reported as missing; can be created in apply mode |
| Source rule missing in target | Reported as drift; can be added in apply mode |
| Target has extra ingress rule | Allowed by default |
| Target has extra egress rule | Allowed by default |
| Target has AWS default outbound rule | Allowed by default |
| Target has all source-required rules | In sync |
| Target tag differs from source | Reported as tag drift |
| Target SG-level description differs from source | Reported as description drift only |
| SG-to-SG reference cannot be mapped | Reported as unresolved dependency |

In simple terms:

```text
The target must contain everything required by source.
The target may contain additional rules.
```

## AWS Access Requirement

Because this version performs a **live comparison**, the machine running the script must have AWS access to both the source and target accounts.

At minimum, the source profile needs read permissions such as:

```text
ec2:DescribeSecurityGroups
```

The target profile needs read and write permissions such as:

```text
ec2:DescribeSecurityGroups
ec2:CreateSecurityGroup
ec2:AuthorizeSecurityGroupIngress
ec2:AuthorizeSecurityGroupEgress
ec2:CreateTags
```

If tag sync is used, the target profile must be allowed to create or update EC2 tags.

## Recommended First Run: Dry Run

Always start with dry-run.

```bash
python sg_reconcile_engine.py \
  --info-file sg-config.json \
  --dry-run
```

Dry-run mode:

- Does not modify AWS.
- Reads source SGs live.
- Reads target SGs live.
- Shows what would be created or updated.
- Builds a Terraform-like plan.
- Writes a JSON report under `sg_reports/`.
- Does not run post-apply validation because no apply operation occurred.

Expected dry-run report behavior:

```json
"post_apply_validation": {
  "enabled": false,
  "status": "NOT_RUN",
  "passed": null,
  "reason": "Post-apply validation only runs after --yes apply mode."
}
```

## Report Only Mode

Use report-only when you want analysis without execution behavior.

```bash
python sg_reconcile_engine.py \
  --info-file sg-config.json \
  --report-only
```

Report-only mode:

- Reads source SGs live.
- Reads target SGs live.
- Builds the matching and drift report.
- Does not create SGs.
- Does not add rules.
- Does not sync tags.
- Does not run post-apply validation.

## Apply Mode

Use apply mode only after reviewing the dry-run report.

```bash
python sg_reconcile_engine.py \
  --info-file sg-config.json \
  --yes
```

Apply mode can:

- Create missing target SG shells.
- Refresh newly created SGs before rule sync.
- Build a source SG ID to target SG ID map.
- Remap source SG references to target SG references.
- Add missing source-required ingress rules.
- Add missing source-required egress rules.
- Sync tags.
- Write rollback action records.
- Run post-apply validation after remediation completes.

After apply mode completes, the tool automatically performs post-apply validation. It re-fetches the target security groups, rebuilds the plan, and verifies whether missing groups, drifted rules, and unresolved dependencies are now cleared.

A successful apply should produce:

```json
"post_apply_validation": {
  "enabled": true,
  "status": "PASSED",
  "passed": true
}
```

If remediation does not fully converge, the report shows:

```json
"post_apply_validation": {
  "enabled": true,
  "status": "FAILED",
  "passed": false
}
```

Apply mode does **not** remove extra target rules under the current source-required-only policy.

## Matching Logic

The engine supports the following source-to-target SG matching methods:

1. **Canonical name match**
   - Best and safest match.
   - Example: source `app-sg` matches target `app-sg`.

2. **CloudFormation tag match**
   - Uses CloudFormation logical ID and stack-name tags when available.

3. **Optional normalized-name fallback**
   - Enabled with `--allow-legacy`.
   - Helps when names differ due to environment prefixes, suffixes, or generated naming patterns.
   - Ambiguous normalized matches are blocked in apply mode.

Example:

```bash
python sg_reconcile_engine.py \
  --info-file sg-config.json \
  --dry-run \
  --allow-legacy
```

Use `--allow-legacy` carefully. If multiple target SGs normalize to the same value, the tool marks the match as ambiguous and blocks apply mode.

## Missing SG Creation Behavior

When a source SG does not exist in the target VPC, apply mode creates an empty SG shell first.

This sequence is intentional:

```text
1. Match existing source and target SGs.
2. Create missing target SG shells.
3. Build source SG ID -> target SG ID mapping.
4. Refresh newly created target SGs.
5. Rewrite SG-to-SG rule references.
6. Apply missing ingress and egress rules.
7. Sync tags.
8. Re-read target SGs.
9. Rebuild validation plan.
10. Write report with post-apply validation result.
```

This is required because SG-to-SG rules can only be created safely after all referenced SGs exist in the target VPC.

---

## SG-to-SG Reference Handling

If a source SG rule references another source SG, the tool remaps that reference to the matching target SG ID.

Example:

```text
Source:
my-sg-b allows ingress from my-sg-a
```

The target should become:

```text
Target:
my-sg-b allows ingress from target version of my-sg-a
```

The tool does not copy source-side SG metadata such as source account ID, source VPC ID, VPC peering fields, or peering status into the target rule. It rewrites the rule using clean target-side SG references.

---

## Dependency and Cycle Handling

Security groups can reference each other.

Example:

```text
sg-a references sg-b
sg-b references sg-a
```

This creates a dependency cycle.

The tool handles this by creating SG shells first, including SGs inside dependency cycles, then applying rules after the target ID map is available.

Cycle detection is reported in the Terraform-like plan.

---

## Important Behavior: Extra Target Rules Are Allowed

By default, the tool tolerates extra target ingress and egress rules.

Example:

```text
Source egress: no outbound 0.0.0.0/0 rule
Target egress: all traffic to 0.0.0.0/0
```

Default result:

```text
No drift, as long as all source-required rules exist in target.
```

This is useful for DR environments where the target may contain AWS default outbound rules, DRS-managed rules, or environment-specific rules.

---

## Strict Extra-Rule Visibility

The `--revoke-extra-rules` option can be used to make extra rules visible as drift in the report.

However, in the current source-required-only policy, extra rule revocation is not actively performed. The option is useful for strict visibility, but the apply logic does not remove extra target rules.

Current behavior:

```text
Default:
- Extra target rules are ignored.
- Extra target rules do not count as drift.

With --revoke-extra-rules:
- Extra target rules may appear as drift.
- Extra target rules are still not removed by the current policy.
```

---

## What Counts as Drift

The tool reports drift when:

- A source-required ingress rule is missing from the target.
- A source-required egress rule is missing from the target.
- Tags differ.
- SG-level descriptions differ.
- A source SG is missing from the target.
- A rule references another source SG that cannot be mapped to a target SG.

The tool does **not** report drift only because the target has additional rules, unless strict extra-rule visibility is enabled.

Note: SG-level description differences are reported for visibility, but AWS does not support safely updating the security group object description in place. This tool does not recreate SGs just to change descriptions.

## Reading the Report

Reports are written to:

```text
sg_reports/
```

Example:

```text
sg_reports/sg_reconcile_dryrun_dr_us-east-1_20260517_035234.json
```

Important report fields:

| Field | Meaning |
|---|---|
| `source_total` | Number of source SGs found |
| `evaluated` | Number of SGs evaluated after exclusions |
| `strict_matches` | SGs matched by strict/canonical logic |
| `legacy_matches` | SGs matched by normalized-name fallback |
| `missing` | Source SGs not found in target |
| `drifted` | SGs with missing source-required rules, tag drift, description drift, or unresolved rule issues |
| `unresolved_dependencies` | SG references that could not be mapped |
| `in_sync` | SGs compliant with the current policy |

A clean result looks like this:

```json
{
  "source_total": 4,
  "evaluated": 4,
  "strict_matches": 4,
  "legacy_matches": 0,
  "missing": 0,
  "drifted": 0,
  "unresolved_dependencies": 0,
  "in_sync": 4
}
```

## Post-Apply Validation

When `--yes` is used, the report includes a `post_apply_validation` section.

| Field | Meaning |
|---|---|
| `enabled` | Whether post-apply validation ran |
| `status` | `PASSED`, `FAILED`, or `NOT_RUN` |
| `passed` | Boolean result of the validation |
| `summary.missing` | Missing SGs after apply |
| `summary.drifted` | Drifted SGs after apply |
| `summary.unresolved_dependencies` | Unresolved SG references after apply |
| `summary.create` | SGs still requiring creation after apply |
| `summary.update` | SGs still requiring updates after apply |
| `summary.skip` | SGs confirmed in sync after apply |

A clean post-apply result looks like:

```json
{
  "enabled": true,
  "status": "PASSED",
  "passed": true,
  "summary": {
    "missing": 0,
    "drifted": 0,
    "unresolved_dependencies": 0,
    "create": 0,
    "update": 0
  }
}
```

Operators should treat `post_apply_validation.status = PASSED` and `passed = true` as the primary confirmation that apply mode converged.

---

## Terraform-Like Plan

The report includes a Terraform-like plan with this structure:

```json
{
  "summary": {
    "create": 0,
    "update": 0,
    "delete": 0,
    "skip": 4,
    "cycles": 0
  }
}
```

| Field | Meaning |
|---|---|
| `create` | Source SGs missing in target |
| `update` | Existing SGs needing missing source-required rules or tag sync |
| `delete` | Reserved for future delete behavior |
| `skip` | SGs already in sync |
| `cycles` | Dependency cycle groups detected |

## Execution Report

The report also includes execution details:

```json
{
  "execution": {
    "created_sgs": [],
    "failed_nodes": [],
    "sg_id_map": {},
    "rollback_actions_count": 0
  }
}
```

| Field | Meaning |
|---|---|
| `created_sgs` | Target SGs created during apply mode |
| `failed_nodes` | Source SGs that failed during creation or rule sync |
| `sg_id_map` | Source SG ID to target SG ID mapping |
| `rollback_actions_count` | Number of rollback actions recorded |

In dry-run mode, no AWS resources are created.

## Troubleshooting

### Missing SG was created but rules did not apply

Likely causes:

- The source SG references another SG that could not be mapped.
- The target profile lacks `ec2:AuthorizeSecurityGroupIngress` or `ec2:AuthorizeSecurityGroupEgress`.
- The rule contains cross-VPC, peering, or cross-account metadata that requires manual review.

Check:

```text
post_apply_validation.status
post_apply_validation.summary.drifted
execution.failed_nodes
sg_plan.unresolved_dependencies
```

### `UnauthorizedOperation`

The target profile does not have the required EC2 write permissions.

### `InvalidVpcID.NotFound`

The target VPC ID is wrong, belongs to another region, or belongs to another account.

### `InvalidGroup.Duplicate`

A security group with the same name already exists in the target VPC. The tool attempts to detect and reuse the existing SG.

### Extra AWS default outbound rule appears in target

This is expected. Extra target egress rules are tolerated by default and should not count as drift in source-required-only mode.

### Post-apply validation fails

The apply action completed, but the target did not fully converge. Review:

```text
post_apply_validation.summary
post_apply_validation.sg_plan.drifts
post_apply_validation.sg_plan.unresolved_dependencies
execution.failed_nodes
```

## Current Limitations

This engine is stable for source-required-only SG reconciliation, but it is still a stabilization-stage tool.

Known limitations:

- It does not delete target SGs.
- It does not actively remove extra target ingress or egress rules.
- It does not perform full automatic rollback.
- It does not perform advanced delete safety checks.
- It does not implement CIDR translation between source and DR subnet ranges.
- It does not deeply validate all AWS service-managed SG use cases.
- Complex cross-account, cross-VPC, or peered-VPC SG references may require manual review.
- Description differences are reported, but AWS does not allow safely updating a security group object description in place.

---

## Best Practices

- Always run dry-run first.
- Review JSON reports before apply mode.
- Use post-apply validation as the primary confirmation after `--yes`.
- Use exact SG names where possible.
- Use `--allow-legacy` only when necessary.
- Do not use `--allow-unresolved-dependencies` unless you understand the missing references.
- Keep test runs small before using this against a large enterprise environment.
- Do not commit real account IDs, VPC IDs, SG IDs, CIDRs, or production SG names into a public repository.
- Keep `sg-config.json` out of public repositories if it contains real environment details.
